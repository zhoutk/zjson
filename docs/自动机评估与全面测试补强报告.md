# zjson 自动机评估与全面测试补强报告（第 4 轮）

## 1. 任务目标与结论

本轮目标：

1. 评估 `src/zjson.hpp` 在 JSON 解析阶段是否需要引入“自动机模式”并给出可实施方案。
2. 在不破坏现有 API 易用性的前提下，增强解析正确性与测试覆盖。
3. 补充性能与内存安全方面的工程化保障手段。
4. 复审你在 `docs/优化评审与改造报告.md` 中已有优化是否合理。

**结论（先给结论）：**

- **当前阶段不建议直接把主解析器重写为完整自动机（DFA/PDA）**。
- 更优策略是：**保留现有递归下降主干 + 增加严格模式入口 + 系统化 conformance 测试 + 可选 ASan 检测**。
- 该策略已落地代码并通过回归测试（8 个测试目标全部通过）。

## 2. 自动机模式是否“必须”

## 2.1 技术判断

从 JSON 语法性质看：

- 词法层（数字、转义）可以用 DFA；
- 结构层（对象/数组嵌套）天然是下推自动机（PDA）问题；
- 解析器常见实现是“递归下降（等价于隐式栈）”或“显式栈状态机”。

你当前实现已经具备：

- 明确的深度上限（`max_depth`）防止无限递归；
- 越界读取防护和错误位置上报；
- O(1) 追加与 O(1) key 查找（`lastChild` / `keymap`）后，性能瓶颈已大幅缓解。

因此，**“是否必须自动机”答案是：不必须**。原因：

1. **收益/风险比不优**：整套重写风险高，容易引入语义回归；
2. **现状性能可用**：Release 解析吞吐已处于较高水平（见第 6 节）；
3. **你的核心诉求是 API 简单好用（add 构建）**，不是极限流式解析器。

## 2.2 何时再上自动机

满足以下条件时再考虑第二阶段：

- 需要 SAX/流式解析超大文档（>100MB）且内存受限；
- 需要可证明的严格状态迁移表（审计/合规要求）；
- 需要 SIMD/分阶段解析（类似 simdjson 体系）追求 GB/s 级吞吐。

## 3. 对既有优化的复审结论

你在上一份报告中的大方向是正确的，尤其以下项非常关键且合理：

1. **`lastChild` 消除尾插 O(n²)**：显著提升大对象解析吞吐；
2. **惰性 `keymap`**：高频 key 访问从线性退化问题中解耦；
3. **拷贝/赋值资源语义修复**：避免泄漏和空指针风险；
4. **字符串转义完整化**：对 RFC 8259 兼容性提升明显；
5. **错误位置（line/col）上报**：可维护性与定位效率显著提升。

可继续关注的 trade-off：

- `keymap` 与 `lastChild` 让节点体积增大，可能轻微影响 `toString` / deep copy 缓存局部性；
- 对你这个“构建+解析并重”的库来说，这个 trade-off 是可接受的。

## 4. 本轮已实施改造（代码已落地）

## 4.1 解析模式改造：严格模式 API

文件：`src/zjson.hpp`

新增/调整：

1. `parse(const std::string&, std::string&, bool allowComments = true)`
2. `parse(const char*, std::string&, bool allowComments = true)`
3. 新增公开入口：`Json::ParseJsonStrict(const std::string&, std::string&)`
4. `JsonParser` 新增字段：`bool allowComments`
5. `consume_comment()` 在严格模式下直接报错：`comments are not allowed in strict mode`

效果：

- 兼容旧行为：`ParseJson` 仍保留扩展模式（支持注释）；
- 新增严格行为：`ParseJsonStrict` 符合 RFC 主流预期（不接受注释扩展）。

## 4.2 工程化内存检查入口

文件：`CMakeLists.txt`

新增：

- 选项 `CMAKE_ZJSON_ENABLE_ASAN`（默认 OFF）
- 非 MSVC 工具链下自动添加 `-fsanitize=address -fno-omit-frame-pointer`

说明：

- 这是“发现泄漏/越界”的工程入口，不改变生产默认行为。

## 4.3 新增系统化 RFC/边界测试

新增文件：`tests/test_rfc_cases.cpp`

新增测试覆盖：

1. **严格模式有效输入**（标量、容器、数字指数、Unicode 代理对、空键名、空白符）
2. **严格模式无效输入**（尾逗号、非法数字、非法转义、非法空白、注释、尾随垃圾）
3. **扩展模式注释用例**（同输入在 strict 下应失败）
4. **重复 key 行为固化**（当前实现 `operator[]` 为 first-wins）
5. **压力回环用例**（parse→copy/move→toString→reparse 循环）

并在 `CMakeLists.txt` 中新增测试目标：

- `test_rfc_cases`

## 5. 全网资料调研与测试策略映射

本轮调研来源：

1. RFC 8259（JSON 标准）
2. Parsing JSON is a Minefield
3. JSONTestSuite（`y_ / n_ / i_`）
4. JSON_checker（PDA 验证器）
5. nativejson-benchmark（conformance + time + memory + alloc 指标方法）

映射到本仓测试策略：

- **结构类**：括号闭合、尾逗号、尾随垃圾、空白符边界
- **数字类**：前导零、小数指数完整性、`NaN/Infinity`
- **字符串类**：非法转义、Unicode 代理对、控制字符
- **扩展类**：注释在 strict 与 extension 下分流验证
- **行为类**：重复 key 可预期行为固化

注：

- JSONTestSuite 的 `i_`（实现定义）用例并非“对错题”，在本库中建议按“文档化行为”而非“一刀切判错”处理。
