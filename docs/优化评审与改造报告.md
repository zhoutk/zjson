你的目标是“简单好用、靠 add 接口覆盖绝大多数 JSON 构建场景”。

## 2. 发现的问题（评审清单）

### 2.1 错误逻辑 / 高风险缺陷

1. **拷贝构造存在空指针解引用风险**
   - 原实现在 `origin.child == nullptr` 时仍会解引用 `*origin.child`。
2. **赋值运算符使用 placement new，未释放旧资源**
   - `operator=` 直接 `new (this) Json(...)`，会导致旧 `child` 链资源泄漏风险。
3. **解析器多处数组越界读**
   - `consume_whitespace`、`parse_number` 等路径对 `str[i]` 访问缺少 `i < size` 防护。
4. **字符串序列化转义不完整**
   - 仅处理了反斜杠，未正确处理 `"`、控制字符、换行/制表等，可能生成不合法 JSON。

### 2.2 设计与可维护性问题

1. **私有死代码存在**
   - `addValueJson/addSubJson` 已不再被调用，保留会增加维护负担。
2. **工具链设置冲突噪声**
   - MSVC `/MTd` 与 gtest 的 `/MDd` 产生冲突告警。

### 2.3 性能与行为一致性问题

1. **字符串序列化构建路径有不必要临时对象**
   - 原 `replaceAll` 路径会产生额外复制。
2. **`test_more` 依赖运行目录**
   - 用例对当前工作目录敏感，CI/不同生成目录下不稳定。

## 4. 性能专项评估（解析与字符串化）

## 4.1 解析性能

本轮主要做了“安全与正确性增强”，对热点路径影响很小；
解析性能瓶颈依然主要在：

- 链表结构导致的局部性不足
- 数值/字符串分支较多
- 目前不是流式解析（一次性输入）

## 4.2 字符串化性能

本轮优化点：

- 减少不必要中间字符串替换
- 改为逐字符追加转义

后续可继续优化：

1. 预估输出容量并 `reserve`
2. 数值格式化统一改为 `to_chars`（可在 C++17/编译器可用性评估后启用）
3. 提供紧凑模式/美化模式分离（目前仅紧凑）

---

## 5. 自动机模式是否有必要

结论：**当前阶段“不是必须”，可作为二阶段增强，而非第一优先级。**

原因：

1. 现有实现是递归下降 + 手写词法处理，已能覆盖主流输入；
2. 你的核心诉求是“简单 add 构建”，不是高吞吐流式解析器；
3. 自动机（尤其 PDA）在“校验/流式解析/超大输入防栈溢出”方面更有价值，但会显著提高实现复杂度。

建议路线：

- **阶段1（已做）**：先把语义正确性与内存安全打牢。
- **阶段2（可选）**：新增“严格模式 DFA/PDA 校验器”，作为独立入口，不替换现有 API。
- **阶段3（可选）**：引入事件流（SAX）解析通道服务大文件。
