## 2. 发现的问题（评审清单）

### 2.1 错误逻辑 / 高风险缺陷

1. **拷贝构造存在空指针解引用风险**
   - 原实现在 `origin.child == nullptr` 时仍会解引用 `*origin.child`。
2. **赋值运算符使用 placement new，未释放旧资源**
   - `operator=` 直接 `new (this) Json(...)`，会导致旧 `child` 链资源泄漏风险。
3. **解析器多处数组越界读**
   - `consume_whitespace`、`parse_number` 等路径对 `str[i]` 访问缺少 `i < size` 防护。
4. **字符串序列化转义不完整**
   - 仅处理了反斜杠，未正确处理 `"`、控制字符、换行/制表等，可能生成不合法 JSON。

### 2.2 设计与可维护性问题

1. **私有死代码存在**
   - `addValueJson/addSubJson` 已不再被调用，保留会增加维护负担。
2. **工具链设置冲突噪声**
   - MSVC `/MTd` 与 gtest 的 `/MDd` 产生冲突告警。

### 2.3 性能与行为一致性问题

1. **字符串序列化构建路径有不必要临时对象**
   - 原 `replaceAll` 路径会产生额外复制。
2. **`test_more` 依赖运行目录**
   - 用例对当前工作目录敏感，CI/不同生成目录下不稳定。

## 4. 性能专项评估（解析与字符串化）

## 4.1 解析性能

本轮主要做了“安全与正确性增强”，对热点路径影响很小；
解析性能瓶颈依然主要在：

- 链表结构导致的局部性不足
- 数值/字符串分支较多
- 目前不是流式解析（一次性输入）

## 4.2 字符串化性能

本轮优化点：

- 减少不必要中间字符串替换
- 改为逐字符追加转义

后续可继续优化：

1. 预估输出容量并 `reserve`
2. 数值格式化统一改为 `to_chars`（可在 C++17/编译器可用性评估后启用）
3. 提供紧凑模式/美化模式分离（目前仅紧凑）

## 5. 自动机模式是否有必要

结论：**当前阶段“不是必须”，可作为二阶段增强，而非第一优先级。**

原因：

1. 现有实现是递归下降 + 手写词法处理，已能覆盖主流输入；
2. 你的核心诉求是“简单 add 构建”，不是高吞吐流式解析器；
3. 自动机（尤其 PDA）在“校验/流式解析/超大输入防栈溢出”方面更有价值，但会显著提高实现复杂度。

建议路线：

- **阶段1（已做）**：先把语义正确性与内存安全打牢。
- **阶段2（可选）**：新增“严格模式 DFA/PDA 校验器”，作为独立入口，不替换现有 API。
- **阶段3（可选）**：引入事件流（SAX）解析通道服务大文件。

### 9.1 第一轮做对的部分

| 改进项 | 评价 |
|--------|------|
| 新增 `cloneChain` / `releaseChildren` 辅助函数 | 思路正确，解决了拷贝构造空指针问题 |
| 重写 `operator=`，移除 placement new | 正确修复了赋值运算符资源泄漏 |
| 解析器 `parse_number` / `consume_whitespace` / `consume_comment` 边界检查 | 必要且正确 |
| 字符串值转义（`appendEscapedString` lambda） | 方向正确，补全了 RFC 8259 要求的 7 种转义 |
| `FromFile` 改为 `ios::binary` + `istreambuf_iterator` | 消除了逐行读取丢失换行的 bug |
| 清理死代码（`addValueJson`、`addSubJson` 等） | 正确，降低维护负担 |

### 9.2 第一轮遗留缺陷（本轮修复）

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | **`Json(const char*)` 仍使用 `new(this)` placement new** | 高 | GPT 在 `operator=` 中批判并移除了 placement new，但在 `const char*` 构造函数中遗留了同样的反模式。改为 `*this = Json(string(jsonStr))` 安全委派 |
| 2 | **JSON key 名（键名）序列化未转义** | 高 | `valueJsonToString` 和 `toString` 中 key 名直接拼接，若 key 含 `"`、`\`、控制字符则输出非法 JSON。抽取 `appendQuotedKey` 统一处理 |
| 3 | **`operator[]` 整数下标：空数组时解引用 nullptr** | 高 | `this->child->find(index)` 未检查 `child == nullptr`，空数组下标访问直接崩溃 |
| 4 | **`stringEndWith` 字符串短于 tail 时无符号溢出** | 中 | `str.size() - tail.size()` 下溢，传入 `compare` 的偏移量为天文数字 |
| 5 | **`sprintf` 残留（两处）** | 中 | `%e` 与 `%lld` 格式化仍用 `sprintf`，改为 `snprintf` |
| 6 | **`operator=(Json&&)` 缺 `noexcept`** | 中 | 移动构造已加 `noexcept`，移动赋值遗漏，导致 STL 容器不能调用移动赋值 |
| 7 | **`cloneChain` 对 brother 链递归** | 中 | 宽对象（>数千 key）复制时递归深度 = key 数量，有栈溢出风险。改为 brother 链迭代 + child 递归 |
| 8 | **`deleteJson(nullptr)` 崩溃** | 低 | do-while 至少执行一次，对 nullptr 参数直接解引用 |
| 9 | **`add` 方法运算符优先级不明** | 低 | `!name.empty() && type==Object \|\| type==Array` 缺少显式括号，可读性差 |

## 10. 第二轮已实施改造

### 10.1 `src/zjson.hpp`

1. **`stringEndWith`** — 增加 `str.size() < tail.size()` 前置判断
2. **新增 `appendEscapedString(input, out)` 命名空间级帮助函数** — 提取自 lambda，供 key/value 双方使用
3. **新增 `appendQuotedKey(key, out)` 帮助函数** — 输出 `"escaped_key":`
4. **`Json(const char*)` 构造** — 移除 `new(this)`，改为 `*this = Json(string(jsonStr))`
5. **`cloneChain`** — brother 链改为迭代，child 保持递归（深度受 `max_depth` 控制）
6. **`operator=(Json&&)` 加 `noexcept`**
7. **`operator[]` int 版** — 增加 `this->child == nullptr` 检查
8. **`deleteJson`** — 增加 `if (!obj) return` 前置守卫
9. **`valueJsonToString`** — 移除 lambda；全部 5 处 key 输出改用 `appendQuotedKey`；`sprintf` → `snprintf`
10. **`toString`** — Object/Array 子节点 key 输出改用 `appendQuotedKey`
11. **3 个 `add` 重载** — 补充显式括号 `(!name.empty() && ...) || ...`

### 10.2 `tests/test_spec.cpp`

新增 4 个测试用例：

- `key_name_escaping` — key 含引号/反斜杠/换行/制表符，验证序列化与回环解析
- `key_escaping_for_nested_objects` — 嵌套对象的 key 转义（走 `toString` 的 `appendQuotedKey` 路径）
- `empty_array_subscript_access` — 空数组 `[0]`/`[1]`/`[-1]` 不崩溃
- `wide_object_copy` — 2000 key 宽对象拷贝，验证迭代式 `cloneChain` 不栈溢出

### 11.1 错误位置上报（line/column）

### 11.2 性能基准

- 已完成 `bench/` 目录，使用 `std::chrono` 计时（不引入第三方依赖）
- 测试数据集：~4KB / ~10KB / ~44KB / ~100KB / ~1MB JSON
- 指标：parse 吞吐（MB/s）、toString 吞吐（MB/s）、深拷贝吞吐（MB/s）

### 11.3 已实施 — 错误位置上报（line/column）

- `JsonParser` 新增 `make_error()` 方法，懒计算 line/col
- `fail()` 输出格式：`"line X, col Y: <message>"`
- 新增公开 API：`Json::ParseJson(input, errMsg)` 返回解析结果并填充错误信息
- 新增测试用例 `parse_error_reports_position`

### 11.4 数据结构升级（已实施）

在保留链表结构的前提下，新增两个字段消除了最主要的两个热点瓶颈，API 完全不变：

**`Json* lastChild`（O(1) 尾追加）**
- 所有容器节点维护指向最后一个子节点的尾指针
- `appendNodeToJson` 由 O(n) 变为 O(1)，彻底消除解析 O(n²) 行为
- `cloneChain` 重构为带 `outTail` 参数的版本，复制时无需二次扫描

**`mutable std::unordered_map<string, Json*>* keymap`（O(1) key 查找）**
- 惰性构建：首次 `operator[](string)` 或 `contains()` 时触发 O(n) 一次性建表
- 后续同一容器的所有 key 查找均 O(1)
- 修改操作（`appendNodeToJson` / `remove` / `clear` / `releaseChildren`）自动清除缓存，下次查询重建

**关联更新**：`push_front`、`remove(int)`、`removeFirst`、`insert(int,...)` 均已同步维护 `lastChild`。
