# ZJSON 性能测试报告

> 测试时间：2026-05-26  
> 环境：Windows，CMake `x64-release` preset，MSVC Release `/O2`，`bench_zjson`。  
> 对照库：zjson、nlohmann/json、RapidJSON、simdjson。  
> 代码范围：节点池 / 字符串存储改造后，含解析器重复键检测优化。

## 本次改造内容

- 节点分配：为 heap `Json` 节点增加 `thread_local` slab allocator，保留原有 `new Json(...)` / `delete` 语义，避免解析和深拷贝时大量小对象直接打到通用堆分配器。
- 字符串存储：将内部 `name` / `valueString` 从直接 `std::string` 改为 `detail::StoredString`，解析时由 `StringArena` 持有源缓冲；无转义字符串保存为 `std::string_view`，有转义字符串在 arena 内物化，公开 API 仍返回/接受 `std::string`。
- 解析器重复键检测：对象帧内使用固定小数组保存前 16 个键，宽对象再升级到 `unordered_map<string_view, Json*>`，避免默认 `KeepLast` 策略对宽对象造成 O(n²) 扫描。
- Benchmark：`bench/bench_main.cpp` 已升级为多库 harness；本次实际编译并运行了 nlohmann/json、RapidJSON、simdjson。

新增覆盖重点：

- 解析源字符串销毁或覆写后，key/value 仍有效。
- 有转义字符串大量解析后，arena 中物化字符串引用保持稳定。
- 解析后执行 `remove` / `mergePatch` / `add` 等修改不产生悬垂引用。
- 大量创建、复制、清空树结构，覆盖 slab allocator 的复用路径。

## Comparative Benchmark

说明：

- throughput 单位均为 MB/s，数值越高越好。
- simdjson 当前在本 benchmark 中只统计 parse；它不是通用可变 DOM + copy/stringify 同口径实现，因此未参与 stringify/copy 对照。
- 数据集 `flat_*` 偏宽对象，`mixed_*` 偏混合对象/数组嵌套，更接近日常 JSON 负载。

## 结果解读

- `zjson` parse 在全部数据集上都明显快于 nlohmann/json，优势约为 1.4x 到 2.3x；宽对象 `flat_100KB` 达到 125.42 MB/s，说明本轮重复键检测优化已经解决此前宽对象退化问题。
- `zjson` copy 也稳定快于 nlohmann/json，优势约为 2.1x 到 2.5x，说明 slab allocator 对节点复制/释放路径是有效的。
- stringify 方面，`zjson` 目前落后于 nlohmann/json 和 RapidJSON，尤其在 mixed 数据集上差距更明显。这部分主要仍受链式节点遍历和字符串输出路径影响，不是本轮内存优化的主要收益面。
- RapidJSON 作为通用 DOM 库在三类操作上都保持显著领先，simdjson 在 parse 上则进一步拉开差距；这与它们更激进的内存布局和 SIMD 解析设计一致。

## 结论

- 本轮“节点池 + arena-backed 字符串存储 + 解析器对象键索引”改造，已经把 `zjson` 的 parse/copy 拉到明显优于 nlohmann/json 的区间，并消除了宽对象场景的严重退化。
- 当前 `zjson` 的主要短板已从“解析分配和宽对象重复键处理”转移到 stringify 路径。若继续推进性能路线图，下一优先级应放在序列化路径的遍历/输出优化，而不是继续堆内存层面的微调。
- 以本次 Windows/MSVC Release 数据为准，`zjson` 更适合作为“比 nlohmann/json 更轻、更快的单头文件 DOM 方案”，但目标暂时不是追平 RapidJSON/simdjson 这类为性能深度特化的实现。

后续建议：

- 序列化路径继续做结构化 profiling，重点看 `toString` 的链表遍历和字符串拼接热点。
- 若要做更公平的 simdjson 对照，可增加“parse only”单独图表，避免与可变 DOM 的 copy/stringify 混在同一指标里。
